# i18n — pattern & progress

StudyOS ships with `next-intl` configured for **runtime locale swap, no URL routing**.
The user's locale comes from `user_preferences.preferred_language` — chosen during
onboarding — so signed-in users get their language on every render without any
`/en/...` or `/pa/...` URL segments.

## The pattern

**Server Components** (default):

```tsx
import { getTranslations } from "next-intl/server";

export default async function SomePage() {
  const t = await getTranslations("notes");
  return <h1>{t("title")}</h1>;
}
```

**Client Components** (`"use client"` at top):

```tsx
import { useTranslations } from "next-intl";

export function SomeInteractive() {
  const t = useTranslations("auth.login");
  return <button>{t("cta")}</button>;
}
```

**Rules**

1. Never inline a user-facing English string. Route it through a `t()` call.
2. One namespace per component. `useTranslations("landing")` — not `useTranslations()` with global lookup — so keys stay short and typos surface early.
3. Interpolate values with `{name}` placeholders, resolved by `t("welcome", { name })`. Don't string-concat.
4. Add new keys to **both** `en.json` and `pa.json` before shipping. `hi.json` stays behind until `LOCALE_METADATA.hi.released = true`.
5. Nav labels use `labelKey` (typed union) — see `components/layout/nav-config.ts`. The renderer resolves it with `useTranslations("nav")`. Never hardcode label strings on `NavItem`.
6. In auth-shell forms, the shell (`AuthShell`) is a Server Component and reads
   its own `shellHome` label; the form component owns its own translations.

## Namespace map

| Namespace | Owner | Used by |
| --- | --- | --- |
| `common` | Shared verbs / CTAs | Any surface (`t("save")`, `t("cancel")`, `t("signOut")`) |
| `errors` | Generic error messages | Toasts, error boundaries |
| `nav` | Bottom nav / sidebar labels | `nav-config.ts` + both renderers |
| `landing` | Marketing home | `LandingExperience`, `AudienceToggle` |
| `auth` | Sign-in / sign-up / shell | `SignInForm`, `SignUpForm`, `AudiencePicker`, `AuthShell` |
| `dashboard` | Home greeting | `GreetingHeader` |
| `profile` | Settings + role chip | `/app/profile` |
| `notes` | Notes list + empty state | `/app/notes`, sidebar "New note" CTA |
| `community` | Feed + empty state | `/app/community`, `CommunityEmptyState` |
| `onboarding` | Wizard | Onboarding steps |

## What Module 13 shipped

- Landing page fully translated (`LandingExperience`, `AudienceToggle`).
- Nav labels translated across mobile bottom nav + desktop sidebar.
- Auth pages: `/login`, `/signup`, both marketing panels' Home link, `AuthShell` titles/eyebrows, `SignInForm`, `SignUpForm`, `AudiencePicker`.
- Dashboard: greeting header (morning / afternoon / evening).
- Profile page: title, role chip, appearance card, settings rows, version line.
- Notes list header + "New note" CTA (both mobile aria-label + desktop button).
- Community list header + empty state.

Everything above renders correctly in both `en` and `pa`. Switch via
onboarding language step or by manually flipping `user_preferences.preferred_language`.

- AI Quizzes (creator form, quiz runner, results).

- Toast messages from every server action's error path.
- Zod validation messages (`enter your email`, etc.) — these live in schemas and need `zod-i18n-map` or a translated-error factory.

## Adding a new locale

1. Add the code to `SUPPORTED_LOCALES` in `lib/i18n/config.ts`.
2. Create `lib/i18n/messages/{code}.json` with the same shape as `en.json`.
3. Add an entry to `LOCALE_METADATA` — set `released: true` when the dictionary is complete.
4. Add a matching row in the `mediums` table if the locale also doubles as a study medium (Punjabi / Hindi / etc. are both).

## Testing

- Manually: log in → onboarding → pick Punjabi as interface language → land on dashboard. Every surface above renders in Punjabi.
- Regression: `pnpm exec tsc --noEmit` catches missing `useTranslations` imports because the resulting `t` variable is `never`.

## Why no URL-based locales?

- Every user is authenticated to see the app shell. Locale is a per-user preference, not a URL fact — putting it in the URL creates two representations of the same page and complicates deep-linking.
- SEO: only the public landing at `/` matters for search. That page can add
  `<link rel="alternate" hreflang>` metadata in a future pass without moving to URL-based routing.

See ADR-0017 for the full trade-off.
